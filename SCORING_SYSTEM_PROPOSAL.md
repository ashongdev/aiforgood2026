# RFGYC Grand Finale — Digital Scoring System

## Technical Proposal & Operational Overview

---

## 1. System Overview

### Brief Description of the Scoring System

This system is a purpose-built, live tournament management platform designed specifically for the RFGYC Grand Finale. It replaces the previous spreadsheet-based pipeline with a fully integrated web application backed by a dedicated cloud database, enabling real-time score entry, instant public display, and granular administrative control — all without requiring any spreadsheet access or formula knowledge from field staff.

The system operates in two distinct layers working in parallel:

| Feature                 | What the System IS NOT                                                                                     | What the System IS                                                                                                                     |
| ----------------------- | ---------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| **Match Officiating**   | It is NOT a digital referee or automated judge. It does not replace the traditional human referee process. | It IS completely dependent on human referees who strictly supervise every match and write scores on paper.                             |
| **Score Tracking**      | It does NOT replace manual scoring sheets.                                                                 | It IS a dedicated data-entry interface where credentialed scorekeepers type points directly from official paper sheets.                |
| **System Purpose**      | It is NOT a standalone tracking system that runs the physical game.                                        | It IS a digital, live-updating tournament board designed to display calculations and matchups cleanly and instantly.                   |
| **Audience Visibility** | It is NOT a hidden internal tool meant only for judges.                                                    | It IS a visual window bridging the gap between the field and the venue, letting spectators, coaches, and teams follow the action live. |

### The Two Layers

- **The Backend Data Layer (Supabase / PostgreSQL):** A secure, cloud-hosted relational database that stores all team names, match records, and scores. It enforces strict access control at the database level — no unauthorised user can write data even if they bypass the UI. Scores written by scorekeepers are instantly available to all connected clients without any polling delay.

- **The Frontend Presentation Layer (React Web App):** A clean, responsive web application split into two experiences: a **public spectator view** (read-only leaderboard and bracket display) and a **private scorekeeper interface** (authenticated data entry screen). Both update in real time the moment any score is saved.

### Purpose and Scope for the RFGYC Grand Finale

The sole purpose of this application is to run the Grand Finale smoothly, quickly, and without human calculation errors.

- **The Scope:** The system manages everything from the moment teams arrive until trophies are handed out. It covers the **Qualifying Phase** (where all teams take multiple rounds to achieve a best score across all rounds regardless of total score), automatically filters the top teams by rank, and feeds them into the knockout elimination rounds: **Pre-Quarterfinals, Quarterfinals, Semifinals, Third Place, and the Grand Finals**. Bracket seeding ensures matches are perfectly balanced (1st place vs. last qualifying place).

## 2. User Roles and Access Levels

To keep the tournament secure and prevent accidental changes, the system enforces strict role-based access at every layer — both in the application interface and directly at the database level via Row-Level Security (RLS) policies.

### Administrator Access

Administrators are the master users who control the entire system. They log in to a dedicated Admin Dashboard that is completely separate from the public site. They can:

- Add, edit, or remove teams (including bulk CSV import)
- Create and manage scorekeeper accounts (generate credentials, assign table numbers)
- Lock or unlock any phase for spectators or for scorekeepers independently
- Suspend individual scorekeepers from score entry
- View a full audit trail of every score change, filterable by phase, scorekeeper, team, or category
- Confirm match winners and advance teams between elimination rounds

### Scorekeeper Access

Scorekeepers are field staff stationed at game tables. They receive login credentials (email + password) generated by the administrator before the event. Once logged in, they see only the Score Entry interface. They:

- Can type raw scores into their assigned match rows
- Cannot access the Admin Dashboard or any administrative controls
- Cannot see or modify other scorekeepers' access
- Are immediately blocked (in real time, without needing a page refresh) if an administrator suspends their account or locks the current phase
- Can operate offline — scores typed during an internet outage are queued locally and automatically synced the moment connectivity is restored

### Spectator / Public Access

Teams, coaches, parents, and the general audience access the public-facing web application on any device without logging in. This view is:

- 100% read-only — there is no input field or pathway to alter any data
- Live-updating via real-time push (no manual refresh required)
- Automatically replaced with a "Phase Locked / Results Pending" screen when the administrator activates the spectator lock for a given phase

## 3. Scoring Workflow

The workflow is built to be fast and simple so that scorekeepers can submit numbers in seconds during the live, high-pressure environment of the Grand Finale.

### Step-by-Step Scoring Process

1. **Accessing the System:** Scorekeepers navigate to the application URL on any internet-connected device and log in using the email and password generated for them by the administrator prior to the event. No Google account, app installation, or technical knowledge is required.

2. **Locating the Match:** The scorekeeper selects their active phase (e.g., Qualifiers, Quarterfinals) and category (Junior / Senior) from the dropdown selectors at the top of the screen. They then optionally filter by their assigned table number to narrow the view to their specific rows.

3. **Data Entry:** As soon as a match finishes and scores are collated from the referee's paper sheet, the scorekeeper types the raw points directly into the correct round column for that team. The interface supports keyboard navigation (Tab, Enter, Arrow keys) to move between cells rapidly without using a mouse.

4. **Instant Save & Update:** The moment the scorekeeper presses Enter or clicks out of a cell, the score is immediately written to the cloud database. There is no "Submit" or "Save" button. The public spectator board updates within milliseconds via real-time push — not a timed polling interval.

### How Scores are Entered, Saved, and Updated

- **Data Entry:** Scores are typed directly into the open cells of the Score Entry grid. The interface validates input inline — only integers are accepted; text entries are silently rejected at the input level.
- **Saving:** Saving is fully automatic and happens the instant focus leaves a cell. A scorekeeper can never lose data if their device restarts, because any unsynced scores are held in a local offline queue and replayed as soon as the connection is re-established.
- **Updating / Corrections:** To correct an error, the scorekeeper simply clicks back into the cell, types the new number, and tabs out. The live standings and public board update globally within milliseconds. Every change is recorded in the audit trail with a timestamp and the scorekeeper's identity.

### Handling of Corrections or Disputes

If a team challenges a score or a judge needs to make an official correction:

1. An administrator reviews the audit trail in the Admin Dashboard, which shows the full history of changes for that match — including who made each change and at what time.
2. If the correction is approved, the administrator (or an authorised scorekeeper if the phase is still open) overwrites the disputed value in the Score Entry grid.
3. The live standings on the public display update instantly with the corrected figure.
4. The correction itself is logged in the audit trail for full transparency and record-keeping.

## 4. Scoring Criteria Configuration

### How Scoring Criteria are Defined and Managed

- **Database-Driven Structure:** Each match stores up to four individual round scores per team (R1, R2, R3, R4) as raw integer fields. The application calculates cumulative totals and live running totals client-side from these raw values, so no formula maintenance is required.
- **Administrator-Controlled:** Scoring rules (e.g., how many rounds count, what constitutes a valid score) are configured by the administrator through the Admin Dashboard and enforced at the database schema level. Field scorekeepers cannot alter any formula or calculation logic.

### Weighting System

- **Peak-Score Seeding Priority:** In accordance with the official competition rulebook, the qualification ranking engine prioritises each team's single Best Round Score over their cumulative tournament points to establish primary leaderboard seeding.
- **Automated Tie-Breaking Weights:** If best scores are equal, the system automatically uses the cumulative Total Points across all rounds as the tie-breaker. This logic runs instantly without any human intervention.

### Column-Based Customisation

The architecture uses an open round-column structure (R1, R2, R3, R4). Adding, removing, or repurposing round columns is a configuration-level change that does not require any code deployment.

## 5. Real-Time Scoring and Results

### Live Scoring Display

- **Public Web App:** The React application serves as the clean, public-facing leaderboard display. It is optimised for large venue screens, projectors, and mobile devices of all sizes.
- **Real-Time Push (Not Polling):** Unlike previous spreadsheet-based implementations that refreshed every 60 seconds, this system uses Supabase Realtime — a WebSocket-based push channel. The moment a scorekeeper saves a score, every connected spectator screen receives the update within milliseconds automatically, with no manual browser refresh required.
- **Offline Resilience:** If the spectator's internet drops temporarily, the public interface gracefully continues displaying the last known leaderboard. It does not crash or show an error page.

### Leaderboard Generation

- **Instant Client-Side Calculation:** Standings are recalculated immediately in the browser each time a score update arrives via the real-time channel. No server-side job or scheduled task is required.
- **Qualifier Leaderboard:** Ranks all teams by best round score (primary) then cumulative total (tie-breaker), updated live as scores come in.
- **Elimination Bracket:** Displays confirmed match winners as they are locked in by the administrator, automatically showing the progression path to the Finals.

## 6. Delay / Locking Rules

The system implements two independent locking mechanisms, each serving a distinct operational purpose.

### Spectator Lock (Public View Blackout)

**Purpose:** To hide score data from the public audience during a sensitive scoring window — for example, while verifying final qualifier results before the live bracket reveal.

**Workflow:**

1. **Lock Off (Normal Operation):** During active play, the public view displays live scores as they are entered. The audience and teams can follow the action in real time.
2. **Lock On (Blackout):** At the administrator's discretion, they activate the Spectator Lock for a specific phase. This immediately replaces the public leaderboard or bracket for that phase with a uniform "Phase Locked / Results Pending" screen on all connected devices. Scorekeepers can continue entering and verifying scores behind the locked screen while the audience sees nothing.
3. **Lock Off (Live Reveal):** Once the event organisers are ready for the live main-stage announcement, the administrator removes the Spectator Lock. The finalised results are instantly revealed to all connected venue screens and mobile devices simultaneously.

**Lock Types:**

- **Full Lock:** Hides all scores and team names for the locked phase.
- **Scores Hidden:** Shows team matchups and pairings but hides the numeric scores.

### Scorekeeper Lock (Score Entry Freeze)

**Purpose:** To permanently close score entry for a completed phase, preventing any further modifications once a round is officially concluded.

**Behaviour:** When a Scorekeeper Lock is applied to a phase, any scorekeeper who navigates to that phase sees the full match data in read-only mode. All score cells are replaced with non-editable display fields. A clear "Phase Locked" banner is shown at the top of the screen explaining that score entry is closed. The database also enforces this at the server level — even a direct API call from a locked phase is rejected.

### Personal Scorekeeper Suspension

**Purpose:** To immediately remove a specific individual's ability to enter scores without affecting others.

**Behaviour:** When an administrator suspends a scorekeeper account, that scorekeeper's screen immediately transitions to a full-screen "Access Suspended" overlay (delivered via real-time push, without requiring a page refresh or re-login). The scorekeeper cannot interact with any part of the Score Entry interface until the administrator reinstates them. This lock is also enforced at the database level.

### Lock Authority

The power to activate any lock (Spectator, Scorekeeper Phase, or Personal Suspension) is strictly restricted to the Administrator. Field scorekeepers have no access to any locking control.

## 7. System Architecture Overview

### High-Level Architecture

The system uses a clean three-tier architecture:

- **Frontend (React Web App):** Built with React 19, TypeScript 5.8, Vite 6, and Tailwind CSS v4. This is the visual layer that all users interact with. It does not store data itself — it reads from and writes to Supabase through a secure client SDK.

- **Backend (Supabase / PostgreSQL):** A dedicated cloud-hosted PostgreSQL database that acts as the single source of truth for all teams, matches, scores, and user accounts. All business logic constraints (who can write what, when) are enforced here via Row-Level Security policies that cannot be bypassed from the frontend.

- **Auth Layer (Supabase Auth):** Handles all authentication using industry-standard JWT tokens. Scorekeeper credentials are provisioned by the administrator through the Admin Dashboard and are revokable at any time.

- **Realtime Layer (Supabase Realtime):** WebSocket channels that push database change events to connected clients instantly. Used for: live score updates on the public board, real-time lock state changes pushed to scorekeepers, and instant suspension enforcement.

- **Edge Functions (Deno / Supabase):** Serverless functions that handle privileged operations requiring service-role access (e.g., creating new scorekeeper accounts in the auth system). These run in Supabase's managed cloud and are never exposed to the client.

### Hosting Environment

- **Frontend:** Hosted on a global CDN (Vercel). The page loads instantly on any phone, tablet, or venue screen. No local servers are required on-site.
- **Database & Auth:** Hosted on Supabase's managed cloud infrastructure (backed by AWS), with automatic backups, point-in-time recovery, and 99.9% uptime SLA.

### Security Model

All data access is governed by **Row-Level Security (RLS)** policies at the PostgreSQL level:

- Scorekeepers can only read/write match rows during unlocked phases and while their account is active.
- Admins can read all rows across all tables.
- Public users (unauthenticated) can only read matches and teams — they cannot write anything.
- Phase locks and personal suspensions are enforced at the RLS layer, meaning no UI bypass is possible.

## 8. Connectivity and Offline Handling

### Internet Dependency

- **Active Connection for Score Sync:** Because the database is cloud-hosted, scorekeepers need an active internet connection (Wi-Fi or mobile data) to sync scores in real time.
- **Lightweight Traffic:** Each score submission is a tiny JSON payload (typically under 200 bytes), meaning the system operates flawlessly on congested venue Wi-Fi networks and basic mobile data connections.

### Offline Scoring Capability

- **Local Queue:** If a scorekeeper's device loses internet connectivity, the Score Entry interface detects this immediately and displays an "Offline" banner. Any scores entered during the outage are held in a local queue stored on the device.
- **Automatic Sync:** The moment internet connectivity is restored, the queued scores are automatically flushed to the database in order. The scorekeeper sees a live sync progress indicator and confirmation when all pending changes have been saved.
- **No Data Loss:** A scorekeeper can never lose data due to a connectivity drop. The offline queue persists even if the browser tab is closed and reopened (stored in localStorage).
- **Public View Safety:** If the public display screen loses internet, it continues showing the last known leaderboard state. It does not crash or show an error.

## 9. Security and Integrity Controls

### User Access Provisioning

All scorekeeper accounts are created exclusively by the administrator through the Admin Dashboard before the event begins. The process is:

1. The administrator enters the scorekeeper's email address and (optionally) their assigned table number.
2. The system generates a secure 14-character random password automatically.
3. The credentials are displayed once for the administrator to communicate to the scorekeeper. Optionally, they can be sent via email.
4. No account provisioning is performed on the fly during live tournament hours by anyone other than the administrator.

**Pre-Event Deadline:** All scorekeeper accounts should be created and tested at least 24 hours before the event opening so credentials can be verified and any issues resolved without tournament-day pressure.

### Data Encryption

All data transmitted between the scorekeeper's device, the Supabase cloud servers, and the public React frontend is encrypted in transit using HTTPS (TLS 1.2+). Data at rest is encrypted by Supabase's cloud infrastructure by default.

### Anti-Tampering Measures

- **Row-Level Security:** Every table in the database has RLS policies that restrict exactly what each authenticated user can read or write. These policies are enforced by the database engine — they cannot be disabled by the application or circumvented via direct API calls.
- **Phase Lock Enforcement:** When an administrator locks a phase for scorekeeper entry, the database UPDATE policy for the `matches` table rejects any write to a row in that phase, regardless of where the request originates.
- **Personal Lock Enforcement:** When an administrator suspends a scorekeeper, the database UPDATE policy rejects any match write from that user's JWT, server-side.
- **Read-Only Public Interface:** The public spectator web app has no input fields and makes no authenticated requests. The Supabase public (anon) key used by the public view has only SELECT permissions on the matches and teams tables.
- **Audit Trail:** Every score change is recorded in the `score_audit_log` table by a database trigger. This log captures the scorekeeper's user ID and email, the timestamp, the phase and category, the team names involved, and the exact before/after values for every modified field. The log is append-only and cannot be modified or deleted by scorekeepers.

## 10. Testing and Demo Environment

### Validation Before the Event

The system is verified before competition day through a structured set of test scenarios:

- **Scenario A (Tie-Breaking Logic):** Enter identical best round scores for two teams. Verify the leaderboard correctly promotes the team with the higher cumulative total.
- **Scenario B (Phase Threshold):** Add more than 8 teams to Qualifiers and verify the system activates the Pre-Quarterfinals bracket automatically.
- **Scenario C (Lock Enforcement):** With the Scorekeeper Phase Lock active, attempt to submit a score via the Score Entry interface and confirm the UI blocks it and the database rejects it.
- **Scenario D (Offline Queue):** Disconnect from the internet, enter several scores, then reconnect. Verify all scores sync to the database in the correct order and the sync banner clears.
- **Scenario E (Suspension Real-Time):** While a scorekeeper is actively on the Score Entry screen, have the administrator suspend their account. Verify the "Access Suspended" overlay appears on the scorekeeper's screen within 2 seconds without a page reload.
- **Scenario F (Spectator Lock Reveal):** Activate the Spectator Lock for Qualifiers. Verify the public view shows the lock screen. Enter and finalise all scores. Remove the lock and verify all results appear simultaneously on the public view.
- **Scenario G (Audit Trail Accuracy):** Make a deliberate score correction and verify the audit trail entry captures the original and corrected values, the scorekeeper's email, and the correct timestamp.

## 12. Contingency / Backup Plan During Live Event

## 14. Operational Specifications

### 1. System Capacity and Concurrent Users

The Supabase backend safely supports thousands of concurrent connections. During this tournament, a maximum of 4–8 scorekeepers will be writing data simultaneously, and hundreds of spectators may be reading the public view concurrently. This load is well within standard operating limits and will not cause conflicts, lag, or data drops.

### 2. Mobile Device Responsiveness

The public React web application is fully mobile-responsive. When coaches, teams, or spectators check standings on their phones, the leaderboard grid and bracket cards automatically adapt to fit smaller screens cleanly without horizontal scrolling.

### 3. Concurrent Score Write Handling

If two scorekeepers accidentally attempt to update the exact same cell simultaneously, the database applies a "last-write-wins" rule — the final write committed to the server becomes the official value. Both entries are logged in the audit trail. To prevent conflicts in practice, scorekeepers are strictly assigned to specific tables and match rows so their workflows never overlap.

### 4. Team Match Call-outs

The software automatically updates bracket progressions as winners are confirmed, but it does not send push notifications or text alerts to teams. Calling teams to the competition floor is a manual process handled by the venue Pit Marshall and the Master of Ceremonies using the main microphone system.

### 5. Administrative Lock Authority

The power to activate any phase lock (spectator blackout or score entry freeze) and to suspend individual scorekeepers is strictly restricted to Administrator accounts. Field scorekeepers have no access to any locking, suspension, or administrative control.

### 6. Team Roster Import Format

Teams can be imported in bulk via CSV upload in the Admin Dashboard. The CSV requires only the following columns:

| Column       | Required | Description             |
| ------------ | -------- | ----------------------- |
| `team_name`  | Yes      | The team's display name |
| `category`   | Yes      | `Junior` or `Senior`    |
| `country`    | No       | Country of origin       |
| `coach_name` | No       | Lead coach name         |

A single-column list of team names (one per line) is also accepted if only names are available at import time. All other fields can be filled in later via the team editor.

## 15. Future Roadmap

The current system is production-ready for the Grand Finale. The following upgrades are planned for subsequent editions:

### Historical Data Warehouse

A multi-year data archival layer will be established, permanently storing full match records, team rosters, and round-by-round score metrics across seasonal events, enabling year-over-year performance analysis.

### Advanced Bracket Scalability

While the system currently supports 8-team and 16-team elimination brackets natively, future updates will add support for 32-team brackets, double-elimination formats, and round-robin group stages — all configurable through the Admin Dashboard without code changes.

### Automated Team Notifications

Optional real-time push notifications (via web push or SMS) to team representatives when they are called to the floor or when their next match bracket is confirmed.

### Enhanced Analytics Dashboard

A dedicated analytics panel for administrators showing scoring velocity, inter-table score variance, and flagged anomalies (e.g., an unusually high score entered significantly faster than average) to support integrity monitoring.

### Multi-Event Tournament Management

Support for managing multiple tournaments simultaneously under one admin account, with separate team pools, bracket structures, and category configurations per event.
